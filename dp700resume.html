Comprehensive Course Summary - Interactive Edition

📊 What is Apache Spark? ▼
Definition: Distributed data processing framework for large-scale analytics across multiple processing nodes (Spark pool in Fabric)
Languages Supported: PySpark, Scala, Spark SQL
Real-World Example: Retail company analyzing sales data from multiple stores simultaneously
🏗️ Spark Pool Architecture ▼
  • Driver Program: Coordinates distributed processes
  • Cluster Manager: External service for acquiring and allocating resources
  • Worker Nodes: Execute data processing tasks via executor processes
Pool Types:
  • Starter Pools: Always on, no wait time
  • Custom Pools: Configurable, becomes default for workspace
⚙️ Advanced Configuration ▼
Native Execution Engine: Vectorized processing for Parquet/Delta formats - significantly improves query performance
High Concurrency Mode:
  • Share Spark sessions across multiple notebooks
  • Independent execution via REPL cores
  • Only first notebook is billed for compute
Important: Admins can disable pool customization at Capacity level
📓 Working with Notebooks ▼
Use Cases: Interactive data exploration and analysis
%%pyspark %%scala %%sql
Magic Commands: Specify language using %% prefix
Preferred Data Format: Delta (Delta Lake)
  • Supports transactions, versioning, streaming
  • Relational database features on Spark
📊 Data Visualization ▼
  • Python libraries: matplotlib, seaborn
  • Built-in notebook charts - default table view with chart customization
🏛️ Eventhouse Overview ▼
Purpose: Store real-time event data from streaming sources
Storage: KQL databases with tables queryable via KQL or subset of SQL
Key Features:
  • Automatic partitioning by ingestion time
  • Time-series data optimization
  • Real-Time Dashboards for visualization
  • Fabric Activator for automated actions
🔍 KQL Database Components ▼
  • Tables: Data storage
  • Stored Functions: Encapsulated queries with parameters
  • Materialized Views: Precomputed aggregations, auto-update with new data
  • Shortcuts: External data references
  • Data Streams: Real-time ingestion
Materialized Views: Store precomputed results + delta of new data = faster queries
Join Optimization: Place smaller table first to reduce rows processed
🏞️ Lakehouse Fundamentals ▼
Definition: Database built on data lake using Delta format tables
Combines: SQL analytics (warehouse) + flexibility/scalability (data lake)
Key Features:
  • Schema-on-read format
  • ACID transactions via Delta Lake
  • All data formats supported
  • Auto-scaling & high availability
🏅 Medallion Architecture ▼
Layer Description Purpose
🥉 Bronze Raw layer Data in original format
🥈 Silver Valid layer Cleaned, deduplicated, merged - consistent format
🥇 Gold Enriched layer Business-specific aggregations, ready for analytics/ML
Gold Layer Variations: Multiple gold layers for different teams (finance, sales, data science) or use Data Warehouse as gold layer
🔐 Security Considerations ▼
  • Bronze: Restricted access, read-only for most
  • Silver: Specific member access for transformations
  • Gold: Broad access for consumption teams
⚡ Real-Time Analytics Basics ▼
Events: Records of system activities (clicks, purchases, sensor readings)
Streams: Continuous sequence of events ordered by time
Think of it: Events = digital records, Streams = delivery mechanism
🔄 Solution Components ▼
  1. Data Ingestion: Capture from sources (IoT, CDC, APIs, logs)
  2. Stream Processing: Transform and analyze in-flight
  3. Low-Latency Storage: Fast access storage
  4. Interactive Dashboards: Real-time visualization
  5. Automated Decision Making: Event-driven triggers
📡 Eventstreams ▼
Components:
  • Sources: Where data originates
  • Transformations: Optional processing
  • Destinations: Where processed data goes
🏢 Fabric Data Warehouse ▼
Features: Relational, fully managed, scalable, T-SQL capabilities
Storage: OneLake using Delta Parquet format
📐 Schema Design ▼
Table Type Purpose
Fact Tables Numerical data for analysis
Dimension Tables Descriptive context
🛡️ Dynamic Data Masking (DDM) ▼
Purpose: Obscure sensitive information from non-privileged users
Masking Types:
  • Full: Complete masking
  • Email: Shows first letter + domain
  • Partial: Expose certain characters
  • Random: For numeric data
🔒 Row-Level Security (RLS) ▼
Implementation Steps:
  1. Create filter predicate (inline table-valued function)
  2. Create security policy invoking the function
Best Practices: Separate schemas, avoid type conversions, minimize table joins
📋 Column-Level Security (CLS) ▼
Implementation:
  • Create roles
  • GRANT SELECT on tables
  • DENY SELECT on specific columns
